# JavaDoctor 压力测试报告

> 测试日期：2026-07-13 | 测试工具：Locust 2.45.0 | 目标：模拟 100 人同时使用

---

## 一、测试环境

| 项目 | 配置 |
|------|------|
| CPU | 本地笔记本（Intel/AMD） |
| GPU | 本地单卡（具体型号见系统） |
| 内存 | 16GB+ |
| LLM | Ollama qwen3.5:4b（本地推理，单卡串行） |
| Embedding | Ollama bge-m3（与 LLM 共享 GPU） |
| 数据库 | SQLite（WAL 模式，NullPool） |
| 后端 | FastAPI + uvicorn（单进程，1 worker） |
| 向量库 | ChromaDB 嵌入式 |

---

## 二、测试场景与结果

### 场景 1：API 基准测试（100 用户 / 1 分钟）

> 纯粹 HTTP 短请求：健康检查、注册、登录、CRUD。无 LLM 调用。

| 接口 | 请求数 | 失败 | P50 | P95 | P99 | 评价 |
|------|--------|------|-----|-----|-----|------|
| `GET /api/health` | 1,557 | 0 | 6ms | 200ms | 270ms | 🟢 极快 |
| `POST /api/auth/register` | 100 | 0 | 250ms | 290ms | 300ms | 🟡 bcrypt 瓶颈 |
| `POST /api/auth/login` | 1,015 | 0 | 210ms | 3,600ms | 3,900ms | 🟡 bcrypt 瓶颈 |
| `GET /api/conversations` | 315 | 0 | 17ms | 530ms | 660ms | 🟢 快 |
| `POST /api/conversations` | 315 | 0 | 36ms | 690ms | 800ms | 🟢 快 |
| `DELETE /api/conversations/{id}` | 314 | 0 | 17ms | 430ms | 540ms | 🟢 快 |
| `GET /api/auth/me` | 630 | 0 | 35ms | 530ms | 660ms | 🟢 快 |
| **汇总** | **4,246** | **0** | **65ms** | **1,900ms** | **3,600ms** | **🟢 优秀** |

**RPS：69.4 | 零失败**

> 📌 bcrypt 密码哈希（~200ms/次）在 100 并发注册/登录时成为 CPU 瓶颈，P95 被拉高到 3.6s。生产部署建议用 Redis 缓存 session 减轻登录压力。

---

### 场景 2：闲聊问答（20 用户 / 3 分钟）

> 意图路由识别 → 跳过检索 → 纯 LLM 流式生成。每次耗时 ~3.7s。

| 接口 | 请求数 | 失败 | P50 | P95 | 评价 |
|------|--------|------|-----|-----|------|
| `POST /api/auth/register` | 20 | 0 | 810ms | 1,000ms | 🟡 |
| `POST /api/auth/login` | 20 | 0 | 750ms | 1,700ms | 🟡 |
| `POST /api/conversations` | 20 | 0 | 63ms | 4,300ms | 🟢 |
| `POST /api/chat/{id}` (闲聊) | **2** | 1 | 3,700ms | 3,700ms | 🔴 |
| **汇总** | **63** | **1** | 750ms | 1,700ms | — |

**❌ 仅 2 个聊天请求在 3 分钟内完成**

原因：Ollama 单 GPU 串行推理，20 个用户全部排队等 LLM。每个闲聊耗时 ~3.7s，第 20 个用户需要等前面 19 个全部完成（~70 秒），加上 `wait_time`（8-25s）延迟，大量请求在测试窗口内来不及开始。

---

### 场景 3：技术问答（10 用户 / 5 分钟）

> 完整 RAG 链路：意图路由 → 混合检索（ChromaDB + BM25 + RRF）→ LLM 流式生成。预计每次耗时 20~40s。

| 接口 | 请求数 | 失败 | 说明 |
|------|--------|------|------|
| 注册/登录/创建会话 | 32 | 0 | 仅 on_start 完成 |
| `POST /api/chat/{id}` (技术) | **0** | 0 | 🔴 零完成 |

**❌ 5 分钟内零个技术问答请求完成**

后端日志显示 9 个 `/api/chat` 请求均返回了 HTTP 200，正在流式输出——但 5 分钟测试窗口结束时没有一条完成。单个技术问答（含检索 + 父块回填 + BM25 + LLM 生成 2048 token）预计需要 20~40 秒，10 个用户串行排队 → 最后一人需等待 200~400 秒，已超出 5 分钟测试时限。

## 三、瓶颈分析

```
┌─────────────────────────────────────────────────────────┐
│  瓶颈层级             严重度    影响                      │
├─────────────────────────────────────────────────────────┤
│  🔴 Ollama LLM 推理  致命     单GPU串行，一次只能处理1个  │
│  🔴 bge-m3 Embedding 致命     与LLM抢GPU，检索时互拖     │
│  🟡 SQLite 写锁      中等     WAL模式已缓解，仍串行写    │
│  🟡 bcrypt 密码哈希  中等     200ms/次，100并发相当吃CPU │
│  🟢 ChromaDB 读      较低     嵌入式，本地无网络延迟     │
│  🟢 BM25 关键词检索  较低     进程内，仅CPU无IO          │
│  🟢 JWT 验签         较低     纯CPU，微秒级              │
└─────────────────────────────────────────────────────────┘
```

**核心结论：LLM 推理是本系统不可逾越的瓶颈。** 在本地单 GPU 环境下，无论后端优化多好，聊天类接口的并发能力上限 = 1 个请求在处理 + N 个请求在排队。

## 四、测试中发现的 Bug 与修复

| # | 问题 | 严重度 | 状态 |
|---|------|--------|------|
| 1 | `.env` 中 `REDIS_URL` 指向不存在的 Redis，每次认证请求尝试连接超时 4s | 🔴 | ✅ 已修复 |
| 2 | `cache.py` 连接 Redis 失败后不记忆，每请求重试一次 | 🔴 | ✅ 已修复 |
| 3 | SQLite QueuePool（size=5）在并发下频繁溢出 `QueuePool limit reached` | 🟡 | ✅ 已修复（改 NullPool） |
| 4 | 多个 uvicorn 进程（含僵尸 worker）同时绑定 8000 端口 | 🟡 | ✅ 已修复 |
| 5 | ChromaDB Rust 绑定不兼容 Python multiprocessing（uvicorn --workers） | 🟡 | 📝 已记录 |
| 6 | `busy_timeout=5000` 被误解为延迟，实际是等待上限 | 🟢 | 📝 已解释 |

## 五、优化建议

### 短期（可在当前硬件上做）

| 优先级 | 建议 | 预期效果 |
|--------|------|----------|
| P0 | **换更小的模型**：qwen3:4b 或 qwen3-embedding:4b 之一，减少 GPU 切换 | 减少 embedding/LLM 之间的 GPU 换进换出 |
| P1 | **降低 LLM 生成长度**：`num_predict` 2048 → 1024 | 单次问答时间减半 |
| P1 | **降低上下文窗口**：`num_ctx` 6144 → 4096 | Prompt 编码更快 |
| P2 | **embedding 结果缓存**：`hash(question)` → 检索结果 5min TTL | 热点问题跳过检索 |
| P2 | **闲聊超短回复**：识别人设问题后直接用固定文本回复，不调 LLM | "你是谁"类问题秒回 |

### 中期（需要额外资源）

| 优先级 | 建议 | 预期效果 |
|--------|------|----------|
| P1 | **加 Redis**：session 缓存 + LLM 结果缓存 | 减轻登录压力，热点问题秒回 |
| P2 | **换 PostgreSQL**：用连接池真正支持并发写 | 数据库不再是瓶颈 |
| P3 | **接入云端 LLM API**（通义千问/DeepSeek API） | 彻底解决单 GPU 串行瓶颈 |

## 六、实际承载能力评估

基于测试数据，在**当前硬件**下：

| 场景 | 可承载同时在线 | 可接受体验 |
|------|:---:|------|
| 纯浏览（看历史记录） | **50~100 人** | 页面秒开，CRUD 毫秒级 |
| 偶尔闲聊（"你好"） | **5~10 人** | 排队等待 10~30s |
| 偶尔技术问答 | **3~5 人** | 排队等待 30~90s |
| 密集技术问答 | **1~2 人** | 基本无等待 |
| 100 人同时提问 | **不可用** | 最后一人等 30+ 分钟 |

> ⚠️ 以上是并发提问数，不是在线人数。100 人在线但不同时提问，系统完全可以承受——注册/登录/会话管理/历史查看的 RPS 达到 69.4，足够支撑数百人同时浏览。
