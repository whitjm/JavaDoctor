# JavaDoctor · Java 面试企业级 RAG 知识库问答系统 · 设计文档

> 版本 v1.0 · 2026-07-11
> 基于 LangChain 的企业级 RAG 知识库问答系统，面向 Java 面试场景。

---

## 1. 项目概述

### 1.1 目标
以 LangChain 为核心框架，构建面向 **Java 面试场景**的企业级 RAG 知识库问答系统。用户通过浏览器进行问答，模型回答强制引用知识库原文，帮助求职者解决 Java 基础、JVM、并发、Spring 全家桶、微服务、面试真题、八股文等各类面试问题。

### 1.2 核心特性
- 管理员浏览器端全量知识库管理（上传/删除/编辑/分段/向量维护）
- 强制引用的在线问答，回复底部展示引用片段 + 来源文档 + 页码
- 多用户会话隔离，会话历史全量持久化，随时回看
- 账号体系（注册/登录/改密）+ 角色权限（admin/user）
- 企业级 RAG 优化（混合检索、重排、父子分段、查询改写、语义缓存），循序渐进落地
- 拓展功能：反馈、标题自动生成、数据看板、限流、面试模拟等

### 1.3 技术选型（定稿）

| 层级 | 选型 | 作用 |
|---|---|---|
| 大模型 | Ollama + Qwen2.5:7b | 本地推理，离线答辩 |
| 嵌入模型 | bge-m3 (BAAI) | 1024 维，稠密+稀疏混合 |
| 重排模型 | bge-reranker-v2-m3 | 检索精排（优化阶段引入） |
| RAG 框架 | LangChain + langchain-community ≥0.3 | 编排核心 |
| 后端 | FastAPI (Python 3.11) | 异步 + SSE 流式 |
| 前端 | React18 + TypeScript + Vite | SPA |
| UI | Ant Design 5 | 管理后台组件 |
| 状态管理 | Zustand | 轻量状态 |
| 向量库 | ChromaDB | 语义检索 |
| 关系库 | SQLite(开发) / PostgreSQL(可选) | 用户/会话/历史/元数据 |
| ORM | SQLAlchemy 2.0 + Alembic | 数据访问 + 迁移 |
| 缓存 | Redis | 语义缓存 + 限流 + JWT黑名单 |
| 认证 | python-jose(JWT) + passlib(bcrypt) | 鉴权 + 密码哈希 |
| 文件解析 | LangChain Loaders + PyPDF2/python-docx/openpyxl | 多格式入库 |
| 部署 | Windows 批处理 start.bat | 一键启动前后端 |

## 2. 系统架构

### 2.1 分层架构

```
┌─────────────────────────────────────────────┐
│  React SPA (Ant Design 5 + Zustand)          │
│  登录 / 问答 / 会话历史 / 知识库管理(admin)   │
└───────────────┬─────────────────────────────┘
                │ REST + SSE(流式回答)
┌───────────────▼─────────────────────────────┐
│  FastAPI 网关层                               │
│  ├─ JWT 认证中间件 + 角色校验(admin/user)     │
│  ├─ 限流中间件(Redis)                         │
│  ├─ 账号服务  会话服务  知识库管理服务         │
│  └─ RAG 编排服务(LangChain)                   │
│      检索→混合→Rerank→组装→生成→引用回填       │
└──┬──────────────┬──────────────┬─────────────┘
   │              │              │
┌──▼───┐    ┌────▼─────┐   ┌────▼──────┐
│Chroma│    │SQLAlchemy│   │本地文件   │   ┌───────┐
│(向量)│    │(SQLite)  │   │(原始PDF)  │   │ Redis │
└──────┘    └──────────┘   └───────────┘   └───────┘
                                            (缓存/限流)
        ┌──────────────┐
        │ Ollama       │  ← LLM 推理 + 嵌入服务
        │ Qwen2.5:7b   │
        └──────────────┘
```

### 2.2 双库一致性设计
关系库与向量库通过 `document_id / chunk_id` 关联：
- ChromaDB 每个向量的 metadata 存 `document_id, chunk_id, page_no, source, doc_type`
- 删除文档时，按 `document_id` 批量删除 Chroma 向量 + 关系库记录，保证一致
- 重新入库时，先删旧向量再写新向量，避免脏数据

## 5. RAG 管线设计（需求 7 · 循序渐进）

采用**循序渐进**策略：先跑通基础 RAG（M5），再逐层叠加优化（M7）。每一层作为独立里程碑，前后对比可直接写入论文实验章节。

### 5.1 文档入库管线（离线，管理员触发）
```
上传文件 → LangChain Loader 解析(按类型分发)
  → 文本清洗 → 分段(RecursiveCharacterTextSplitter)
  → bge-m3 嵌入 → 写入 ChromaDB(带 metadata)
  → 分段落库 document_chunks，状态置"已入库"
```
Loader 分发：PDF→PyPDFLoader，DOCX→python-docx，XLSX→openpyxl，TXT/MD→TextLoader，CSV→CSVLoader。

### 5.2 基础 RAG（里程碑 M5，先跑通）
```
用户问题
  → bge-m3 向量化 → ChromaDB Top-K 检索
  → 组装 Prompt(检索片段 + 问题 + 强制引用约束)
  → Qwen2.5:7b 生成(SSE 流式)
  → 引用回填(从检索结果取 source/page)
```

### 5.3 企业级优化层（里程碑 M7，逐个叠加与评测）

| 阶段 | 优化项 | 解决的问题 | 论文对照指标 |
|---|---|---|---|
| 7-a | **混合检索** 向量+BM25，RRF 融合 | 纯语义漏关键词精确匹配 | context recall↑ |
| 7-b | **Rerank** bge-reranker 精排 Top-N | 初排噪声，相关性不够 | context precision↑ |
| 7-c | **父子分段** small-to-big | 小块检索准、大块给上下文全 | faithfulness↑ |
| 7-d | **查询改写** 多轮指代消解 + 多查询/HyDE | 口语化、指代不清 | answer relevancy↑ |
| 7-e | **语义缓存** Redis 相似问命中 | 重复问题降延迟省算力 | 平均响应时间↓ |

### 5.4 强制引用约束（需求 2）
Prompt 强约束：
- 必须**仅**基于提供的检索片段作答，每个论点末尾标注 `[来源n]`
- 检索片段不足以回答时，明确回复"知识库中未找到相关内容"，不允许编造
- 输出结构化引用列表，回填 messages.citations 字段
- 前端底部渲染引用卡片：片段原文 + 来源文档名 + 页码，可点击展开定位

### 5.5 评测（RAGAS）
指标：faithfulness（忠实度）、answer relevancy、context precision、context recall。
构造 Java 面试问答测试集，对基础 RAG 与各优化阶段做量化对比，产出论文实验数据。

## 7. 部署（Windows 批处理一键启动）

不使用 Docker，直接在 Windows 本地以进程方式启动，`start.bat` 一键拉起后端 + 前端。

### 7.1 依赖（宿主机预装，一次性）
- **Python 3.11** + 后端依赖：`cd backend && pip install -r requirements.txt`
- **Node.js 18+** + 前端依赖：`cd frontend && npm install`
- **Ollama**：安装后 `ollama pull qwen3:4b`（本地推理，作为独立服务常驻 11434）
- **Redis for Windows**（可选，用于语义缓存/限流）：未装时后端自动降级为内存缓存
- ChromaDB、SQLite 均为**嵌入式**，随后端进程启动，无需单独服务

### 7.2 start.bat 职责
一个批处理脚本按顺序完成：
1. 检查 Ollama 是否在运行，未运行则 `start "" ollama serve`
2. 新开一个窗口启动后端：`uvicorn app.main:app --port 8000`
3. 新开一个窗口启动前端：`npm run dev`（Vite，默认 5173）
4. 自动打开浏览器到 `http://localhost:5173`

关键实现：用 `start "标题" cmd /k "命令"` 让前后端各占一个可见控制台窗口，日志分开、便于答辩演示和排错。`stop.bat` 则按窗口标题/端口结束对应进程。

### 7.3 start.bat 示例（M1 交付时落地）
```bat
@echo off
chcp 65001 >nul
cd /d %~dp0

echo [1/4] 启动 Ollama...
start "" ollama serve
timeout /t 2 >nul

echo [2/4] 启动后端 FastAPI...
start "JavaDoctor-Backend" cmd /k "cd backend && uvicorn app.main:app --reload --port 8000"

echo [3/4] 启动前端 Vite...
start "JavaDoctor-Frontend" cmd /k "cd frontend && npm run dev"

echo [4/4] 打开浏览器...
timeout /t 5 >nul
start http://localhost:5173

echo 全部启动完成。关闭本窗口不影响服务，停止请运行 stop.bat
```

答辩演示：双击 `start.bat` 一键起全栈，本地离线可用。

## 9. 拓展功能（需求 8）

优先级从高到低：
1. 答案点赞/点踩反馈（沉淀评测数据）
2. 会话标题自动生成（首轮问题摘要）
3. 管理端数据看板（问答量、热点问题、文档统计）
4. 接口限流（Redis，防滥用）
5. 面试模拟追问模式（多轮深挖）
6. 聊天记录导出（Markdown/PDF）
7. 敏感词过滤
