# 单元测试:创建 + 执行 + 报告

给代码里的核心逻辑（一个个函数/零件）写自动化检查程序，跑一遍，核对结果对不对，最后给用户一份成绩单。

用户是编程小白：解释要用大白话（把"函数"说成"零件"，把"断言"说成"核对答案"），报告要说人话。

## 前提:测试工具

本项目是 **Python(FastAPI) 后端 + React(Vite) 前端** 的 RAG 知识库问答系统。

- **后端 Python 逻辑**：统一用 **pytest**。核心可测逻辑集中在 `backend/app/rag/`（文本清洗、混合分段、意图路由、混合检索、引用编号过滤）和 `backend/app/core/`（密码哈希、JWT）。
- **前端 React 逻辑**：如需测前端，用 **Vitest**（和 Vite 同源）。

第一次运行本技能时，先检查是否已安装：
1. 后端：`python -c "import pytest"`。若没有，`pip install pytest`。
2. 前端（仅当要测前端）：看 `frontend/package.json` 有没有 `vitest`，没有则 `npm install -D vitest`。
3. 依赖下载失败时用国内镜像重试。

## 操作步骤

### 第一步:确定要测什么

1. 如果用户指定了某个文件/功能，就测它；没指定就先看项目结构，挑**纯逻辑、好测**的部分优先。本项目首选后端纯函数：
   - `rag/loaders.py` 的 `clean_text`（文本清洗：去引用编号、连字还原、空格归一）
   - `rag/splitter.py` 的分段逻辑
   - `rag/router.py` 的 `is_chitchat`（意图路由）
   - `rag/retriever.py` 的 `_clip_parent`、RRF 融合打分
   - `rag/bm25.py` 的 `_tokenize`
   - `services/chat.py` 的 `_ThinkFilter`、`_CitationFilter`（流式过滤器）
   - `core/security.py` 的密码哈希/校验、JWT 编解码
2. 用大白话告诉用户这次打算测哪几个"零件"、每个零件负责啥。
3. 避开难测的部分（依赖真实 Ollama 模型、Chroma 向量库、数据库、网络的代码）——必要时用"假数据/替身"隔离，或先跳过并说明原因。**不要在测试里真的去连 Ollama 或加载模型。**

### 第二步:创建单元测试

1. 测试文件统一放在 `backend/tests/` 目录，命名形如 `test_xxx.py`。
2. 每个"零件"写几条检查，覆盖：正常情况、边界情况（空数据、0、负数、跨分片）、异常情况。
3. 用 pytest 写法：函数名 `test_*`，`assert 实际 == 期望`；同类用例可用 `@pytest.mark.parametrize` 批量。测试用中文注释/docstring 写清楚"在测什么"。
4. 若被测模块 import 时会触发重依赖（如 `rag/__init__` 连带加载模型客户端），直接从具体模块文件导入被测函数，避免副作用。

### 第三步:执行测试

1. 在 `backend/` 目录运行 `python -m pytest tests/ -v`。
2. 设置 `PYTHONIOENCODING=utf-8` 避免 Windows 控制台中文乱码。
3. 耗时通常几秒。

### 第四步:给出测试报告

跑完后，用大白话给用户一份成绩单，包含：
- 一共测了多少项，通过多少、失败多少；
- 如果全过：明确说"全部通过，这几个零件干得没问题"；
- 如果有失败：逐条说清哪个零件、测的是什么情况、期望的结果 vs 实际结果，并判断是**代码有 bug** 还是**测试写得不对**，给出修复建议；
- 附上关键的原始输出（失败的那几条），方便核对。

## 注意事项

- 测试是"只读验证"，不改动被测的业务代码；除非确认是代码 bug 且用户同意，才去改业务代码。
- 测试用的临时文件、假数据库文件，跑完清理干净。
- 不要在单测里连真实的 Ollama / Chroma / 数据库；这些用假数据或替身隔离，或明确跳过并说明。
- 不确定某段代码该测什么时，先问用户这个零件"本该有什么行为"，再照着写检查。
